<p>In the default project template <a href="https://github.com/go-kratos/kratos-layout">kratos-layout</a> of the microservices framework <a href="https://github.com/go-kratos/kratos">kratos v2</a>, we use <a href="https://github.com/google/wire">google/wire</a> for dependency injection and recommend that developers use this tool when maintaining projects.</p>
<p>At first glance, wire seems counter-intuitive, leading many developers to not understand why to use it or how to use it (including myself in the past). This article aims to help everyone understand the usage of wire.</p>
<div><h2 id="what">What</h2></div>
<p><a href="https://github.com/google/wire">wire</a> is an open-source dependency injection code generation tool for the Go language. It can generate corresponding dependency injection Go code based on your code.</p>
<p>Unlike other dependency injection tools that rely on reflection, wire can report dependency injection issues during the compilation phase (more precisely, during code generation), rather than waiting until runtime, making debugging easier.</p>
<div><h2 id="why">Why</h2></div>
<div><h3 id="understanding-dependency-injection">Understanding Dependency Injection</h3></div>
<p>What is dependency injection? Why use dependency injection?
<del>Dependency injection is a Java legacy</del> (Just kidding)</p>
<p><a href="https://en.wikipedia.org/wiki/Dependency_injection">Dependency Injection</a> (abbreviated as DI) can be understood as a code construction pattern (a way of writing code) that makes your code easier to maintain.</p>
<p>For many software design patterns and architectural concepts, we often struggle to understand why they require complex gymnastics and strange implementation methods. They usually just provide an example and say that writing this way is elegant and good. Since the derivation process of how this pattern evolved is omitted, we only see the result, making it difficult to understand. Next, let’s try to deduce the entire process to see how and why code evolves into the dependency injection pattern, in order to better understand the significance of using dependency injection.</p>
<div><h4 id="what-is-a-dependency">What is a Dependency?</h4></div>
<p>Here, dependency is a noun, not referring to software package dependencies (like the stuff stuffed in node_modules), but rather the external modules (objects/instances) that a particular module (object/instance) in the software relies on.</p>
<div><h4 id="where-is-it-injected">Where is it Injected?</h4></div>
<p>The dependent module is injected into (i.e., passed as a parameter to) the module when the module is created.</p>

<div><h5 id="boss-how-about-we-change-the-storage-again--we-need-to-add-tests-and-require-mocking">Boss: How About We Change the Storage Again? / We Need to Add Tests and Require Mocking</h5></div>
<p>The boss always has broad ideas. After a couple of days, he wants to switch to Postgres for storage; or he asks you to write some test code for the App, specifically testing the logic inside the interface. Usually, we are reluctant to start another database alongside, so we need to mock the data source part to directly return data for the request handler to use, enabling targeted testing.</p>
<p>What to do in this situation? Change the internal code again? That’s not scientific.</p>
<div><h5 id="programming-to-interfaces">Programming to Interfaces</h5></div>
<p>A crucial idea is to <strong>program to interfaces</strong>, not to implementations.</p>
<p>What does programming to implementations mean? For example, the modified parts in the above example: calling mysqlclient’s exec_sql to execute a SQL statement was changed to calling redisclient’s do to execute a get command. Because each client’s interface design is different, every time the implementation changes, the code must be modified.</p>
<p>The idea of programming to interfaces is completely different. We shouldn’t immediately write code based on what the boss wants. First, we must anticipate that the data source implementation is likely to change, so we should prepare (design) from the start.</p>
<div><h6 id="designing-interfaces">Designing Interfaces</h6></div>
<p>In Python, there is a concept called duck-typing: if it quacks like a duck, walks like a duck, and swims like a duck, then it is a duck. Here, quacking, walking, and swimming are the duck interfaces we agree upon, and if you fully implement these interfaces, we can treat you like a duck.</p>
<p>In our example above, whether it’s a MySQL implementation or a Redis implementation, they both have a common function: using an id to query a piece of data. This is the common interface.</p>
<p>We can define an interface called DataSource, which must have a method called GetById that takes an id and returns a string.</p>

<div><h5 id="dependency-injection-finally-makes-its-entrance">Dependency Injection Finally Makes Its Entrance</h5></div>
<p>Remember what was said earlier about dependency injection? The dependent module is injected into (i.e., passed as a parameter to) the initialization function when the module is created. This pattern exactly allows NewApp to get off work early. We initialize NewRedis or NewMySQL outside and directly throw the resulting DataSource to NewApp.</p>
<p>That is:</p>

<p>Throw that chunk of code that reads the configuration file and initializes Redis into the method that initializes the DataSource.</p>

Therefore, you can continue to extract the code that reads the config outward, making NewRedis take a conf and output a DataSource.</p>

<p>Then, by implementing the DataSource interface, you can change the previous method of reading the configuration file and the method of creating the DataSource to arbitrarily modify your underlying implementation (the implementation of reading the configuration file and which DataSource to use for querying data) without having to change a lot of code each time. This makes your code hierarchy clearer and easier to maintain.</p>
<p>This is dependency injection.</p>
<div><h5 id="problems-with-manual-dependency-injection">Problems with Manual Dependency Injection</h5></div>
<p>The chunk of code above that initializes each instance and then plugs them into the respective initialization methods according to their needs, ultimately constructing the app, is the process of injecting dependencies.</p>
<div><figure><figcaption></figcaption><pre><code><div><div><span>c</span><span> </span><span>:=</span><span> </span><span>GetRedisConf</span><span>()</span></div></div><div><div><span>r</span><span> </span><span>:=</span><span> </span><span>NewRedis</span><span>(</span><span>c</span><span>)</span></div></div><div><div><span>app</span><span> </span><span>:=</span><span> </span><span>NewApp</span><span>(</span><span>r</span><span>)</span></div></div></code></pre></figure></div>
<p>Currently, there is only one DataSource, so writing the injection process manually is still acceptable. But once you have more things to maintain, for example, if your NewApp is like <code dir="auto">NewApp(r *Redis, es *ES, us *UserSerivce, db *MySQL) *App</code> and among them, UserService is like <code dir="auto">UserService(pg *Postgres, mm *Memcached)</code>, forming multi-level dependencies that need to be injected, writing it manually becomes very troublesome.</p>
<p>This is where dependency injection tools like wire come into play—their function is to <strong>help you inject dependencies</strong> by generating code, while the actual dependency instances need to be created (initialized) by yourself.</p>
<div><h2 id="how">How</h2></div>
<p>The main problem with wire is that <del>you can’t learn it by reading the documentation</del>. Anyway, when I first read the documentation, I was completely confused—what is this? What is it supposed to do? But through our previous derivation process, we should roughly understand why to use dependency injection and what role wire plays in it—<strong>helping you inject dependencies</strong> by generating code, while the actual dependency instances need to be created (initialized) by yourself.</p>
<p>Next, it becomes clearer.</p>
<p>First, you need to implement a <code dir="auto">wire.go</code> file, defining the Injector inside.</p>
<div><figure><figcaption></figcaption><pre><code><div><div><span>// +build wireinject</span></div></div><div><div>
</div></div><div><div><span>func</span><span> </span><span>initApp</span><span>() (</span><span>*</span><span>App) {</span></div></div><div><div><span> </span><span>panic</span><span>(</span><span>wire</span><span>.</span><span>Build</span><span>(</span><span>GetRedisConf</span><span>, </span><span>NewRedis</span><span>, </span><span>SomeProviderSet</span><span>, </span><span>NewApp</span><span>))</span></div></div><div><div><span>}</span></div></div></code></pre></figure></div>
<p>Then implement the Providers separately.</p>
<p>After executing the <code dir="auto">wire</code> command,
it will scan the entire project and help you generate a <code dir="auto">wire_gen.go</code> file. If there is anything you haven’t implemented properly, it will report an error.</p>

<div><h3 id="reunderstanding">Reunderstanding</h3></div>
<p>Wait, don’t give up yet. Let’s use <del>magical Chinese programming</del> to explain how to do it.</p>
<div><h4 id="who-participates-in-compilation">Who Participates in Compilation?</h4></div>
<p>The <code dir="auto">initApp</code> method above, officially called Injector, due to the comment <code dir="auto">// +build wireinject</code> on the first line of the file, this wire.go file will only be read by wire. The Go compiler will ignore it during code compilation; the actual file read is the generated wire_gen.go file.</p>
<p>The Providers are part of your code and will definitely participate in the compilation process.</p>
<div><h4 id="what-the-heck-is-an-injector">What the Heck is an Injector?</h4></div>
<p>The Injector is the final result you want—the initialization function of the final App object, which is the <code dir="auto">initApp</code> method in the previous example.</p>
<p>Think of it as going to McDonald’s, entering and seeing the ordering machine, tapping a bunch of items, and finally printing out a receipt.</p>
<div><figure><figcaption></figcaption><pre><code><div><div><span>// +build wireinject</span></div></div><div><div>

<div><h4 id="what-the-heck-is-a-provider">What the Heck is a Provider?</h4></div>
<p>Providers are the methods that create each dependency, such as NewRedis and NewApp in the previous example.</p>
<p>You can think of them as what the McDonald’s staff and kitchen need to do:
The McDonald’s kitchen needs to provide the cooking services for these food items—implement these instance initialization methods.</p>

<p>In wire, there is also the concept of a ProviderSet, which is packaging a group of Providers. Usually, when you order, you are lazy and don’t want to order your Big Mac meal like this: I want a Coke, a pack of fries, and a Big Mac burger; you want to just tap once and get a Big Mac meal. This meal is the ProviderSet, a predefined set of recipes. Otherwise, your order list (the Build in the injector) would become super long, which is troublesome for you and tiring for the staff to look at.</p>
